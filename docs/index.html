Proven Today

One narrow security workflow is real and repeatable.

Ingest

Events reach ClickHouse

Vector accepts syslog, demo traffic, and one real Windows forward lane, then stores normalized events in security.events.

Detect

Suspicious logins produce candidates

The current ruleset is intentionally small and focused on repeated failed login activity across SSH-style and Windows logon telemetry.

Alert

Grafana fires real webhooks

Detection rows in security.alert_candidates drive Grafana alerting, and alert-sink receives firing and resolved payloads.

Demo Flow

Five steps, one story.

01

Logs Enter

Syslog, demo traffic, or Windows events enter Vector.

02

Events Store

NATS buffers the stream and ClickHouse stores normalized records.

03

Rules Evaluate

The detection runner checks for suspicious login bursts every 30 seconds.

04

Candidates Land

Matches are written to security.alert_candidates.

05

Alert Fires

Grafana sends a webhook and alert-sink logs the payload.

Use Case Focus

What Hayabusa is trying to prove right now.

Current Detection Coverage

Suspicious login activity on servers

  • Repeated failed SSH-style login events from syslog traffic
  • Repeated failed Windows logons from one real host lane
  • Endpoint activity visibility based on events already stored

Who It Helps

Technical evaluation before platform expansion

  • Engineers validating a self-hosted telemetry path
  • Consultants who want a credible setup/integration baseline
  • Teams that need a narrow demo without buying into a full SIEM

What It Is

Credible early product foundation

What Is Deferred

Everything outside the proven path

Auth, user management, APIs, custom UI, clustering, compliance, and wider SOC workflows are intentionally deferred until the suspicious-login path is mature enough to justify them.
